2. Privacy Policy

2.1 Data Collection

Anvil does not collect, process, transmit, store, or share any personal data, usage data, analytics, telemetry, crash reports, or any other information about you or your system. The Software runs entirely on your local machine with zero network calls to external services (other than peer nodes you explicitly configure on your own local network).

2.2 Local Data Storage

The following data is stored locally on your machine and never leaves it:

  • Wallet key files (JSON files containing Ed25519 key pairs) stored in the wallets/ directory
  • Blockchain data (blocks, transactions) stored in memory during runtime
  • Node.js module cache (standard npm package files in node_modules/)

No data is persisted to disk beyond wallet key files unless you explicitly implement such functionality.

2.3 Network Communication

When you run the multi-node feature, nodes communicate over HTTP on your local machine (localhost). This communication:

  • Occurs only between nodes you explicitly start on your own machine
  • Uses only the ports you configure (default: 3001, 3002, 3003)
  • Contains only blockchain data (blocks, transactions, peer lists)
  • Does not include any personal information
  • Is not encrypted (HTTP, not HTTPS) since it is local-only
  • Does not connect to any external servers, APIs, or services

2.4 Third-Party Services

Anvil does not integrate with, send data to, or receive data from any third-party services, analytics platforms, advertising networks, or external APIs. The Software has one runtime dependency (Express.js) which is used solely as a local HTTP server and does not perform any data collection.

3.3 Security Limitations

While Anvil uses real cryptographic primitives (SHA-256 for hashing, Ed25519 for digital signatures), the overall system architecture is intentionally simplified for educational purposes. Known security limitations include but are not limited to:

  • No Merkle trees -- transactions are stored as a flat array, not a cryptographic hash tree
  • No UTXO model -- balances are computed by scanning the entire chain, not tracking unspent outputs
  • No mempool prioritization -- transactions are processed strictly first-come, first-served
  • No transaction fees -- only block rewards incentivize mining
  • No difficulty adjustment -- mining difficulty is a static configuration value
  • No account nonces -- replay protection depends on balance checks, not sequence numbers
  • Unencrypted key storage -- private keys are saved as plain-text JSON files
  • HTTP networking -- node communication is unencrypted HTTP, not P2P with TLS
  • No fork resolution -- consensus considers only chain length, not cumulative proof-of-work
  • No Sybil resistance -- no mechanism prevents one actor from running many nodes
  • No eclipse attack protection -- nodes trust any peer that connects
  • No block size enforcement -- the limit is a soft configuration, not consensus-enforced
  • Single-threaded mining -- no parallel hash computation
  • In-memory blockchain -- the chain is lost when the process exits unless explicitly saved

Do not use this Software to secure anything of value. It is designed for learning, not production use.

11. OpenSSL / Cryptographic Notice

Anvil uses the built-in Node.js crypto module for SHA-256 hashing and Ed25519 digital signatures. The Node.js crypto module is powered by OpenSSL (or BoringSSL / quictls depending on the Node.js build).

11.1 Export Compliance

This Software includes cryptographic functionality. Depending on your jurisdiction, the import, export, or use of cryptographic software may be subject to regulation. You are solely responsible for complying with any such restrictions in your jurisdiction. The cryptographic functionality in Anvil is provided by the Node.js runtime, which you install separately.

11.2 Cryptographic Primitives Used

  • SHA-256 (Secure Hash Algorithm 256-bit) -- used for block hashing and transaction hashing
  • Ed25519 (Edwards-curve Digital Signature Algorithm) -- used for transaction signing and verification

Both algorithms are well-established, widely reviewed, and considered secure for their intended purposes. However, as noted in Section 3.3, the system architecture around these primitives is simplified for educational purposes and should not be relied upon for security.
